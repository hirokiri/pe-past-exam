# 分析: 小問題ごとに模範解答および解説を記載

## Source

- Issue: https://github.com/hirokiri/pe-past-exam/issues/15 (#15)
- Slug: `per-subquestion-answers`

## Goal

現状のコンテンツMarkdownは「## 問題（小問(1)〜(n)を含む全文）→ ## 模範解答 → ## 解説」の3セクション構成で、読者は小問と対応する解答・解説を行き来する必要がある。これを **小問題（設問）ごとに「問題文 → 模範解答 → 解説」を連続して記載するスタイル** に変更する。全設問にまたがる解説（出題趣旨・背景知識など）は末尾にまとめて記載してよい（Issue 本文で明示的に許容）。

完了の定義: 小問を含む全コンテンツが新スタイルに変換され、バリデータ・テンプレート・ドキュメントが新スタイルを規定し、`bun run scripts/validate-content.ts`・テスト・ビルドが green で、Web表示・エクスポートが崩れないこと。

## Requirements

### 機能要件

- [ ] 小問（(1)〜(n)）を持つ問題ファイルは、設問ごとに「問題文 → 模範解答 → 解説」を連続して記載する構成に変更する
- [ ] 全設問にわたる解説（出題趣旨・背景知識・出典等）は末尾の全体解説セクションにまとめてよい
- [ ] 小問を持たない問題（II-1-x 等の単一設問）は現行の「問題 → 模範解答 → 解説」構成のまま（既に Issue の意図する「直ちに正答を確認できる」構成を満たす）
- [ ] `scripts/validate-content.ts` が新旧2形式（小問あり/なし）を正しく検証する
- [ ] `scripts/new-question.ts` の雛形が新スタイルを案内する（小問ありの場合の書き方を示す）
- [ ] `docs/PIPELINE.md`（転記・解答作成プロンプト、レビュー観点）と `docs/SPEC.md`（記載の順序）を新スタイルに合わせて更新する

### 非機能要件

- [ ] 問題文の転記内容は改変しない（分割・移動のみ。原文PDFとの一致を維持）
- [ ] 模範解答・解説の記述内容は原則維持し、設問単位への再配置のみ行う（情報の欠落を出さない）
- [ ] frontmatter（id, status 等）は変更しない
- [ ] Web表示（question.tsx / markdown.server.ts はセクション構造非依存でレンダリング）と Vivliostyle エクスポート（TOC は sectionDepth:1 で H1 のみ）を壊さない

## 提案する新フォーマット（小問ありの場合）

```markdown
## 問題

（問題選択の指示・共通リード文・本文のみ。小問(1)〜(n)はここに含めない）

## 設問（1）

（小問(1)の問題文 — 原文のまま）

### 模範解答

（設問(1)に対応する答案部分）

### 解説

（設問(1)固有の解説 — 答案作成のポイント等。無い場合は省略可）

## 設問（2）
…（以下同様）…

## 全体解説

（全設問にわたる内容: 出題趣旨・背景知識など）

### 出典・参考文献
- …
```

- 末尾セクション名は現行の `## 解説` と区別するため `## 全体解説` とする（バリデータで判別しやすい）。
- 現行の 解説 内訳（出題趣旨/答案作成のポイント/背景知識/出典・参考文献）のうち、答案作成のポイントは概ね設問別に書かれているため各設問の `### 解説` へ分配し、出題趣旨・背景知識・出典は `## 全体解説` に残す。
- バリデータの新ルール: `## 設問` 見出しを含むファイルは「`## 問題` → (`## 設問（n）` → `### 模範解答` [→ `### 解説`])+ → `## 全体解説`」順を検証。含まないファイルは従来どおり `問題→模範解答→解説`。`出典・参考文献` 必須チェック（answered 以降）は両形式共通。

## Constraints & assumptions

- **[仮定]** 新フォーマットの見出し構造は上記提案とする。Issue は見出しレベルまで規定していないため、実装物として具体化した（非ブロッキング）。
- **[仮定]** 小問を持たない II-1-x（現在7年×4問中の既存分）は変更対象外。Issue の「小問題ごとに」という表現は小問を持つ問題のみに適用されると解釈。
- **[仮定]** 模範解答は現在1本の論文体で書かれているが、`### N. …（設問(n)）` 見出しで設問対応が明示されているため、機械的+編集的な分割が可能。分割で答案としての流れが不自然になる箇所（結語など）は直近の設問側に付ける。
- **[制約]** 問題文は原文PDFからの忠実な転記であり、語句の改変は不可。リード文と小問の分割のみ行う。
- **[制約]** 42ファイルはすべて `status: reviewed`。再構成後も status は維持する（内容自体は変えないため）。
- 別worktree（worktrees/feature/*）で進行中の機能（nested-export-toc 等）がエクスポートTOCの sectionDepth を変える可能性があるが、本変更は H2/H3 の再編であり H1 は不変のため干渉しない想定。

## Acceptance criteria

1. 小問を含む42ファイルすべてで、各設問の問題文の直後にその設問の模範解答（および設問固有の解説があればそれ）が続く構成になっている。
2. 全設問にわたる解説・出典は末尾の `## 全体解説` に集約されている（出典・参考文献を含む）。
3. 問題文のテキストが再構成の前後で欠落・改変されていない（分割・見出し追加を除き diff で追跡可能）。
4. 模範解答・解説の内容が欠落していない。
5. `bun run scripts/validate-content.ts` が全71ファイルで OK（新旧両形式を検証）。
6. `bun test`・typecheck・`bun run build`（もしくは既存の green 確認手段）が成功する。
7. 小問なしファイル（II-1-x）は変更されていない。
8. `docs/PIPELINE.md`・`docs/SPEC.md`・`scripts/new-question.ts` が新スタイルを規定/案内している。
9. Web アプリの問題詳細ページが新構成で正常に表示される（スモーク確認）。
