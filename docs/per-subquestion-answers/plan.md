# 実装計画: 小問題ごとに模範解答および解説を記載

対応: [analyze.md](./analyze.md) / Issue #15
ブランチ: `feature/per-subquestion-answers`（`worktrees/feature/per-subquestion-answers` の worktree で作業）

新フォーマット（小問ありファイル）の定義は analyze.md「提案する新フォーマット」を正とする:
`## 問題`（共通リード）→ `## 設問（n）`（小問原文）→ `### 模範解答` → `### 解説`（設問固有、省略可）… → `## 全体解説`（出題趣旨・背景知識）→ `### 出典・参考文献`。

## Step 1: バリデータを新旧2形式対応にする

- [x] `scripts/validate-content.ts` を更新
  - 本文に `## 設問` 見出しが含まれる場合の新ルール:
    1. `## 問題` が先頭セクションであること
    2. `## 設問（n）` が1つ以上あり、番号 n が 1 から連番であること
    3. 各 `## 設問（n）` ブロック内に `### 模範解答` が必ずあること（`### 解説` は任意、順序は 模範解答→解説）
    4. 最後のセクションが `## 全体解説` であること
    5. `## 模範解答` / `## 解説`（H2）が残っていないこと（旧形式との混在禁止）
  - `## 設問` を含まないファイルは従来どおり `## 問題 → ## 模範解答 → ## 解説` の順序検証
  - `出典・参考文献` 必須チェック（answered/reviewed）は両形式共通のまま
- 検証: `bun run validate` — この時点では全71ファイルが旧形式のため OK が出ること（新ルールは `## 設問` を含むファイルにのみ発火）。加えて故意に壊したサンプルで NG になることを一時ファイルで手元確認（コミットしない）

## リスク・注意点

- **最大のリスクは Step 4 の編集品質**（転記原文の改変・解答内容の欠落）。機械的置換では済まないため、ファイルごとに丁寧に分割し、バッチごとに検証する。
- 旧→新の切替はコンテンツとバリデータの整合が要。Step 1 の新ルールは「`## 設問` を含むファイルのみ」に発火する設計なので、変換途中でも validate は常に green を維持できる（ロールバックも年度バッチ単位で可能）。
- 並行 worktree（nested-export-toc 等）が `scripts/export.ts` を変更中の可能性があるが、本計画は export.ts に触れないため競合しない。
